# 代码审查治理规则

> 本文定义 `docs/code-review/` 权威增量审查层的方法和质量门，不证明任何源码已经审查或验证完成。证据码、覆盖状态、净汉字与去重阈值均为治理定义，不是源码中的既有约定。

## 1. 适用范围与权威边界

1. 审查对象以覆盖矩阵登记的仓库相对路径为准；每次审查必须绑定 Git 提交和工作树状态。
2. 已提交基线与未提交工作树必须分开描述。结论来自未提交内容时标记 `WORKTREE`，不得归入某个提交。
3. `docs/CODE_REVIEW.md`、`docs/REVIEW_PART2.md` 和 `docs/PROJECT_DOCS_COMPLETE.md` 仅作历史线索；结论只有重新核对、进入问题台账并取得规定证据后，才属于当前审查结论。
4. 治理规则、源码事实、数学结论、运行观察、外部资料和历史线索必须分层表述；不得把计划写成完成状态，把代码存在写成算法正确，把退出码为零写成数值正确。
5. “逐语句块完成”只表示覆盖完整，不表示无缺陷，也不表示 MATLAB 验证通过。

## 2. 基线与审查批次

每批记录：批次 ID、基线提交、分支、工作树是否干净、纳入文件、排除项及理由、审查者、起止时间。源码在审查期间变化时，以文件内容哈希判断：哈希未变可沿用结论；哈希变化则按稳定锚点重定位，不能唯一定位的块标记 `STALE`。

禁止将以下内容混为同一统计口径：

- 文件已登记与文件已审查；
- 物理行、非空行、可执行语句和语句块；
- 静态结论与运行结论；
- 历史稿篇幅与当前有效净汉字；
- 重复解释次数与覆盖次数。

## 3. 逐语句块覆盖

### 3.1 语句块定义

语句块是在同一函数、同一控制流层级内，完成一个可独立说明职责的最小连续单元。遇到下列边界必须切分：

- 脚本段、主函数、每个局部或嵌套函数；
- 函数签名、默认参数、输入校验和输出组装；
- 每个 `if`、`elseif`、`else`，每个 `switch case`、`otherwise`；
- 循环初始化、循环头、循环体、循环后汇总；
- `try` 与各 `catch`；
- `return`、`break`、`continue`、`error` 等提前终止路径；
- 单位转换、坐标转换、随机数、状态更新、协方差更新、文件 I/O、绘图与日志；
- dispatcher 的每个 action；
- 一行内用逗号或分号连接、语义独立的多条语句。

纯空行和视觉分隔符不进入可执行覆盖分母。普通说明注释可随相邻块审查；公式、单位、状态编码、前后置条件及“保证”“唯一”“已验证”等强断言必须单列或明确映射，因为错误注释会形成接口风险。

### 3.2 块级必填项

每个块至少记录：稳定锚点、职责、输入输出及单位、控制流与状态影响、边界或失败路径、结论、证据码、关联 Issue。不得只复述源码。无问题也必须给出结论和证据。

块状态取值：

| 状态 | 含义 |
|---|---|
| `NO_ISSUE` | 已检查，当前未发现问题 |
| `ISSUE:<ID>` | 已检查，并关联一个或多个问题 |
| `N/A` | 不适用；必须写理由，经复核后才计入有效覆盖 |
| `DEFERRED` | 暂缓；必须写原因、责任人或解除条件，不计完成 |
| `STALE` | 源码漂移且无法唯一重定位，不计完成 |

覆盖率按下式计算：

```text
有效覆盖率 = (NO_ISSUE + ISSUE + 经批准的 N/A) / 全部应审查语句块
```

只有有效覆盖率为 100%，且不存在 `DEFERRED`、`STALE`，才可标记文件 `REVIEWED`。同一源码块在多个文档出现只计一次；文本相同但位于不同分支的块分别计数。覆盖矩阵的 `VERIFIED` 还要求关键结论满足第 10 节运行门。

## 4. 证据码

以下为本项目治理层新定义：

| 码 | 名称与定义 | 可以支持 | 不能单独支持 |
|---|---|---|---|
| `S` | Source：指定提交或明确工作树中的源码、配置、仓库文档 | 实现、常量、分支、调用关系确实存在 | 数学正确、路径可达、运行或性能表现 |
| `M` | Mathematics：可复核推导、量纲分析、边界计算或数值证明 | 公式自洽性、理论边界和特定输入结果 | 当前实现确实按该公式执行 |
| `R` | Runtime：在规定环境、命令和输入下实际观察 | 本次执行的行为和输出 | 未覆盖输入、一般正确性或性能改善 |
| `B` | Baseline/Benchmark：与明确基线、参考实现、修复前版本或数值 oracle 的受控比较 | 回归、差异、改善幅度 | 基线自身正确 |
| `L` | Literature：论文、标准或 MathWorks 官方资料 | 外部算法定义、API 语义、物理来源 | 当前实现符合资料 |
| `H` | History：Git 提交、blame、历史分支或提交说明 | 演变过程和历史线索 | 当前行为正确或作者真实意图 |
| `X` | Conflict/Unknown：证据冲突、无法复现、定义不明或未确认 | 明示结论受阻或存在反证 | 任何正向证明；它是阻断标记 |

### 4.1 使用规则

- 组合写作 `[S+M]`、`[S+R+B]`；正文必须能定位每种证据，不写笼统“已验证”。
- 实现事实通常用 `[S]`；数学正确性至少需要 `[S+M]`；运行故障通常至少需要 `[S+R]`。
- 修复后代码变化只能证明“已修改”；确认修复至少需要修复后 `[S+R]`。涉及精度或性能改善时增加 `[B]`。
- 与外部算法一致通常至少需要 `[S+L]`；涉及公式映射时使用 `[S+M+L]`。
- `B` 必须记录比较双方、输入、随机种子、指标和方法；`L`、`H` 不能替代当前源码证据。
- 带 `X` 的结论不得进入 `VERIFIED`。未确认推测写作 `HYPOTHESIS [X]`，不得作为确定缺陷计数。
- 严重度与证据强度独立；高严重度、低证据的问题仍只能处于候选或争议状态。

## 5. 严重度 P0—P3

| 级别 | 定义 |
|---|---|
| `P0` | 阻断或结果不可采用：主要入口普遍不能运行；核心指标系统性失真并足以推翻主要结论；静默保存错误结果；不可规避的数据破坏、安全或合规问题 |
| `P1` | 重要正确性：影响主要算法、入口或常用场景，可导致明显错误、发散、错误关联或不可复现，但有规避方式或影响部分输入 |
| `P2` | 局部正确性、健壮性或显著维护风险：边界失败、错误处理不足、实现漂移、明显性能/内存问题、可测试性差 |
| `P3` | 低影响改进：命名、注释、接口清晰度、格式、轻量重构或不改变结果的微优化 |

P0 的运行性主张原则上需 `[S+R]`；纯数学致命错误可先用 `[S+M]` 确认并尽快补运行复现。重复代码本身不高于 P2；只有已有证据表明它造成更高等级影响时才升级。每个 Issue 另设 `confidence=high|medium|low`。升级、降级均记录旧级别、原因、日期和证据，不静默覆盖。

## 6. Issue 状态机

主流程：

```text
CANDIDATE → CONFIRMED → FIX_PROPOSED → FIXED_UNVERIFIED → VERIFIED
```

旁路状态：

| 状态 | 含义 |
|---|---|
| `DISPUTED` | 定义或证据有争议，必须带 `X` |
| `BLOCKED` | 缺环境、数据或依赖；记录解除条件 |
| `WONT_FIX` | 风险被明确接受；记录负责人、理由和复审条件 |
| `DUPLICATE` | 指向主 Issue；锚点并入主项，不重复计数 |
| `OBSOLETE` | 代码删除或架构变化使问题不再适用，不等于已验证修复 |
| `FALSE_POSITIVE` | 复核证明原结论错误；保留勘误和反证 |
| `REOPENED` | 验证失败或回归重现，重新进入确认流程 |

状态门：`CANDIDATE → CONFIRMED` 至少有 `S`，并按问题性质补 `M` 或 `R`；`FIXED_UNVERIFIED` 只表示源码已变化；源码问题进入 `VERIFIED` 必须有 R2023a 验证记录，纯文档问题可用双人或独立静态复核代替。删除问题代码优先标 `OBSOLETE`，除非已有运行证据证明预期行为恢复。

每项至少包含：ID、标题、严重度、置信度、状态、影响入口、稳定锚点、最小复现或推导、证据、建议、负责人、创建/更新日期、关联验证 ID、重复项和状态历史。

## 7. 行号漂移与稳定锚点

行号只供浏览，不得作为唯一定位。标准格式：

```text
<commit-or-WORKTREE>:<repo-relative-path>
::<function-or-script-section>
::<block-kind>#<ordinal>
| L<start>-L<end>
| first="<规范化首句>"
| last="<规范化末句>"
| hash=<规范化块内容SHA-256前12位>
```

规范化时统一换行和空白，但不删除运算符、字符串、变量名或控制流关键字。重定位顺序为：路径；函数名或 `%%` 节标题；首尾语句；块哈希；前后各两条语句上下文。不能唯一匹配即标 `STALE`，不得把旧结论自动套到相似代码。重命名记录旧、新路径；一项问题涉及多处实现时保存多个锚点，不用超大行号范围掩盖不同职责。

## 8. 净汉字质量门

净汉字指中文技术正文中的汉字数，扣除：代码围栏、行内代码、路径、URL、命令、提交哈希、原样源码、自动目录/表头，以及重复段落第二次及以后出现的汉字。净汉字用于抑制空泛和源码抄录，不能替代覆盖、证据或正确性。

最低门：

- 每个完成审查的非平凡 MATLAB 文件不少于 120 个净汉字；
- 每个 `CONFIRMED` Issue 的问题、影响、证据和建议合计不少于 80 个净汉字；
- 少于 20 行且无 Issue 的简单包装函数可降至 60 个，但仍须完整覆盖表；
- 分支多、公式复杂或问题多的文件不能以达到最低字数为完成理由。

统计必须报告脚本版本、纳入文件、排除规则、原始汉字、扣除项、有效净汉字；不得用文件字节数、总字符数或历史拼接稿代替。

## 9. 去重质量门

先去除 Markdown 标记并规范空白；路径、提交哈希、行号和纯数字可替换为占位符。随后执行精确段落哈希和近重复检测：

- 非模板正文的精确重复段落必须为 0；
- 可重复字段只允许出现在表头或集中模板，不得复制整段结论；
- 两段均不少于 40 个汉字且字符 5-gram Jaccard 相似度 `>=0.85` 时，必须合并，或记录保留理由；
- 扣除批准模板后，唯一净汉字比例必须 `>=0.90`；
- 同一源码块的重复评论不增加覆盖数和净汉字；汇总页应引用 Issue 或审查块，不复制全文。

质量门失败时本批不得宣称达到篇幅或完成目标。历史材料中的重复不追溯计入当前有效字数。

## 10. 函数式过程风格与 OOP 禁令

仓库目标是纯函数式核心、函数式过程组织和显式状态传递；这不等于当前所有旧代码已具备数学意义上的引用透明性。

强制规则：

1. 禁止新增用户定义 `classdef`、`properties`、`methods`、`events` 和 handle class。
2. 领域状态以结构体等显式参数传入，并通过返回值传出；核心数学函数在相同显式输入下应产生相同输出。
3. 禁止新增 `global`；禁止使用 `eval`、`evalin`、`assignin` 或 `run('header.m')` 注入隐式领域状态；禁止依赖调用者工作区中未声明变量。
4. 随机行为显式设置并记录 seed 或 `RandStream`；时间、环境变量和文件内容等外部输入应在入口层取得后显式下传。
5. 文件 I/O、绘图、日志和随机场景生成留在入口或适配层，不混入可独立计算的核心算法。
6. action dispatcher 允许保留，但每个 action 必须有明确输入、输出、状态副作用和错误契约。

`run_*.m`、`scan_*.m` 顶层脚本可承担工作区、绘图和文件输出副作用。MATLAB 内置 `table`、`datetime`、`containers.Map` 和图形句柄不视为建立用户领域 OOP 体系。旧南阳子系统中的 `global` 和 `run('header.m')` 属现存技术债，应登记审查，不得据此降低新增代码约束。

## 11. MATLAB R2023a 验证协议

### 11.1 固定环境

所有 MATLAB 运行证据必须使用：

```bash
"C:\Program Files\MATLAB\R2023a\bin\matlab" -batch "cd('D:/Desktop/single_target_with-turn'); assert(strcmp(version('-release'),'2023a'),'Required MATLAB release is R2023a'); addpath(genpath('.')); <your_code>"
```

不得用其他版本结果替代；其他版本可作补充并标 `X`，但不能获得项目 `R` 通过。验证必须固定工作目录。脚本入口用 `run('入口.m')`，不要假设调用者工作区。

### 11.2 验证层级

1. 静态：`checkcode` 与接口/语法检查；只产生静态辅助证据，不等同于 `R` 行为验证。
2. 单函数：确定输入、断言输出、单位、尺寸、容差、边界与错误路径。
3. 入口冒烟：执行受影响主入口，检查退出码、关键断言和产物内容。
4. 回归/基准：固定输入、参数和随机种子，比较修复前后或 oracle，标 `[R+B]`。
5. 蒙特卡洛/扫描：与快速冒烟分开；记录重复次数、统计指标和接受区间。

一次修改至少运行直接受影响函数及最短真实入口；跨模块契约变化补上下游入口；涉及随机、融合、关联或评估指标时固定种子并做数值断言。没有正式测试套件不能成为免测理由。

### 11.3 运行记录必填项

每条 `R` 至少记录：验证 ID、提交与工作树状态、`version` 和 `version('-release')`、完整命令、工作目录、入口/函数、输入参数、随机种子、起止时间、退出码、关键 stdout/stderr、断言及结果、产物路径和摘要、warning、复现次数、审查者。

判定规则：

- 退出码 0 只证明命令结束；必须有预定断言或指标范围。
- “图看起来正常”和“生成了 MAT 文件”不能单独作为通过证据。
- 修复前后比较必须固定 MATLAB 版本、输入、参数和 seed。
- 缺工具箱、数据或权限导致失败时标 `BLOCKED [R+X]`，不能直接归为源码缺陷。
- 未实际运行不得使用 `R`；未比较基线不得使用 `B`；带 `X` 不得标记通过。

## 12. 批次验收

每批同时报告：文件与语句块覆盖、`DEFERRED/STALE`、各状态 Issue 数、证据分布、R2023a 验证覆盖、净汉字、精确重复数、近重复数、唯一净汉字比例、未解决冲突。任一强制门失败时只能报告实际进度，不得宣称该批完成。

历史结论被推翻时进入勘误记录，并将相关 Issue 标为 `FALSE_POSITIVE`、`DISPUTED`、`OBSOLETE` 或由新证据决定的状态；禁止静默删除。治理规则变更也必须记录日期、理由和对既有统计口径的影响。
